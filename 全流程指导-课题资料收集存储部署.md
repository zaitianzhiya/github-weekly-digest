# 课题资料收集、存储与部署全流程指导

> 基于「GitHub 信息收集整理平台」项目（2026-07-05）的实战经验总结，涵盖从调研、文档撰写、代码开发到 GitHub Actions 部署的完整流程与避坑指南。

---

## 一、信息调研阶段

### 1.1 调研策略：广度优先，再深度挖掘

**正确做法：**

先用多个不同角度的关键词进行广撒网搜索，覆盖中英文不同生态。例如本次调研使用了：

- 英文侧：`"weekly GitHub digest" curated content`、`"AI Safety Digest" methodology workflow`
- 中文侧：`"GitHub 周刊" 整理 方法`、`"科技爱好者周刊" 信息来源 筛选`
- 工程侧：直接 fetch 项目 README 和源码（`raw.githubusercontent.com`）获取未在搜索摘要中呈现的实现细节

**避免的坑：**
- 不要只看搜索结果摘要，关键信息（如评分权重、Prompt 设计、去重策略）往往藏在项目源码和配置文件中
- 中英文生态差异大，分开搜索能发现不同范式的项目

### 1.2 信息源交叉验证

对每个调研对象，至少从三个维度获取信息：

| 维度 | 方法 | 本案例 |
|------|------|--------|
| 官方文档 | 直接读取 README、SKILL.md、配置文件 | `AI Safety Digest` 的完整 README 和 `oranger0611` 的 `prompts/instructions.md` |
| 源码验证 | 查看实际代码实现，确认文档描述的准确性 | `kkkano/tech-digest-daily` 的项目结构、`oranger0611` 的 Provider 检测逻辑 |
| 社区讨论 | 搜索相关 Issue、V2EX 讨论、博客分析 | 阮一峰周刊的 V2EX 讨论、"逆向工程"信息源的帖子 |

### 1.3 结构化记录调研结果

**正确做法：** 调研结果立即写入结构化 Markdown 文档，而非留在对话上下文中。模板：

```markdown
## 项目名

- **信息源**：列举具体来源和采集方式
- **筛选机制**：量化标准还是人工判断
- **技术栈**：语言、框架、调度方式
- **发布渠道**：GitHub/邮件/公众号/网站
- **核心优势**：可复制的设计亮点
- **核心劣势**：已知局限和风险
```

**避免的坑：** 对话中积累的调研信息如果不及时落盘，会话中断后全部丢失。

## 二、文档撰写与合并

### 2.1 多份调研报告的合并策略

本次项目中，Claude 和 Kimi 各自产生了一份报告，合并时遵循的原则：

1. **先评价，后合并**：写一份独立的评价文档，逐条列出两份报告各自的优缺点
2. **互补优先**：识别两份报告的互补关系（本案例中 Claude 报告偏方法论 + AI 策略，Kimi 报告偏工程实现 + 量化评分），合并时保留双方优势
3. **冲突裁决**：当两份报告对同一问题有不同结论时，以有实战数据验证的一方为准；都没有数据时以逻辑推理为准
4. **显式标注来源**：合并后的文档中标注每个模块的来源，方便追溯

### 2.2 文档的版本与归档

**正确做法：**

```
知识库/
├── 01-调研报告/           # 原始调研文档
│   ├── github-weekly-research-report.md      # Claude 调研
│   └── GitHub信息收集整理平台调研...md        # Kimi 调研
├── 02-分析评价/           # 评价与合并文档
│   └── kimi-report-review-and-merge.md
├── 03-最终方案/           # 最终完整版
│   └── GitHub信息收集整理-最终完整方案.md
├── 04-项目代码/           # 实现代码
│   └── github-weekly-digest/
└── 05-经验总结/           # 本文档
    └── 全流程指导.md
```

**避免的坑：** 不要把不同阶段的文档混在一个目录下，后期的实现代码会淹没前期的调研文档。

## 三、项目开发阶段

### 3.1 目录结构设计

**推荐结构（已验证有效）：**

```
project/
├── .github/workflows/     # GitHub Actions 工作流
├── config/                # YAML 配置文件（与代码分离）
├── prompts/               # LLM Prompt 文件（与代码分离，可随时编辑）
├── src/                   # 源代码
│   ├── collectors/        # 各信息源采集器
│   ├── filters/           # 去重、过滤、评分
│   ├── ai/                # LLM 客户端、摘要、分析
│   └── render/            # Markdown 渲染输出
├── data/                  # 运行时数据（gitignore）
├── output/                # 生成产物（gitignore，但 CI 中 force-add）
├── run.py                 # 统一入口（避免相对导入问题）
├── CLAUDE.md              # 项目 AI 助手指南
└── requirements.txt
```

### 3.2 配置分离原则

**核心原则：** 可变的配置与不变的代码严格分离。

| 配置类型 | 文件 | 用途 |
|---------|------|------|
| 信息源 | `config/sources.yml` | 数据源 URL、采集频率、Tier 分层、生态分组 |
| 关键词 | `config/keywords.yml` | 正向加权/负向降权词、重点关注作者名单 |
| 质量阈值 | `config/quality.yml` | 评分权重、可信度等级、过滤规则 |
| AI Prompt | `prompts/*.md` | 摘要/分析/分类的指令（修改后无需重新部署） |

**避免的坑：** 不要把 API Key、Token 硬编码在代码或配置文件中。使用环境变量 + GitHub Secrets。

### 3.3 Python 导入问题（最大坑之一）

**问题：** 相对导入（`from .base import ...`）和绝对导入（`from src.collectors.base import ...`）混用会导致 `ModuleNotFoundError`。

**根因：** `run.py` 在仓库根目录作为脚本运行（`python run.py`），此时 `src` 是一个普通目录而非 Python 包，相对导入会失败。

**正确做法：** 统一使用绝对导入，并在 `run.py` 开头添加 `sys.path.insert(0, '.')`。

### 3.4 网络驱动器文件损坏（最大坑之二）

**问题：** 通过网络驱动器（如 `Z:\` 映射的网络存储）写入文件时，文件内容可能被 null 字节（`\x00`）填充，导致：
- Python 报 `ValueError: source code string cannot contain null bytes`
- YAML 解析失败（`unacceptable character #x0000`）
- GitHub Actions 无法识别 `workflow_dispatch` 触发器

**检测方法：**
```python
with open(filepath, 'rb') as f:
    data = f.read()
    if b'\x00' in data:
        print(f"CORRUPTED: {filepath}")
```

**解决方案：**
1. **对于 Python 源码**：用 bash heredoc 直接写入（`cat > file << 'PYEOF' ... PYEOF`）
2. **对于 YAML 工作流文件**：同上，用 bash heredoc 写入，然后用 `python3 -c` 验证无 null 字节后再提交
3. **定期验证**：部署前对所有 `.py` 和 `.yml` 文件做 null 字节检查

### 3.5 LLM 接入的容错设计

**核心原则：LLM 永远不应该成为系统的单点故障。**

```python
try:
    summary = llm_client.chat(prompt, data)
except Exception as e:
    print(f"AI skipped: {e}")
    summary = ""  # 降级为空，继续渲染数据版周报
```

**实践要点：**
- AI 调用失败时自动降级为纯数据输出，不阻断流水线
- 选择合适的免费模型避免成本问题（Gemini 免费版 → Qwen 欠费 → 最终 Gemini 成功）
- Prompt 与代码分离，放在独立的 `.md` 文件中，修改后无需重新部署

## 四、GitHub Actions 部署

### 4.1 Secrets 配置

**必须配置的 Secrets：**

| Secret | 用途 | 获取方式 |
|--------|------|---------|
| `GH_TOKEN` | GitHub API 访问 + push 权限 | GitHub Settings → Developer settings → Personal access tokens → 勾选 `repo` + `workflow` |
| `GEMINI_API_KEY` | AI 摘要生成 | https://aistudio.google.com/apikey（免费） |

**关键细节：** checkout 步骤必须使用 `secrets.GH_TOKEN` 才能有 push 权限：
```yaml
- uses: actions/checkout@v4
  with:
    token: ${{ secrets.GH_TOKEN }}  # 不加这行，git push 会 403
```

### 4.2 定时任务配置

**Cron 最佳实践：**
- **避开整点**：GitHub 在整点调度负载最高，可能导致任务延迟或丢失。用 `23 8 * * *` 而非 `0 8 * * *`
- **添加并发控制**：防止调度器重复触发
  ```yaml
  concurrency:
    group: daily-digest
    cancel-in-progress: false
  ```
- **添加看门狗**：主工作流可能被 GitHub 调度器丢失，用多个检查点自动补发
- **GitHub Actions 的 schedule 是 "best-effort"，无 SLA 保证**：对此类限制要有预期

### 4.3 工作流文件的完整性

**问题：** YAML 工作流文件含 null 字节后，GitHub 仍然显示文件内容正确（因为 raw.githubusercontent.com 会传输完整内容），但 Actions 引擎解析失败导致 `workflow_dispatch` 触发器不可用。

**验证方法：**
```bash
# 推送前检查
grep -c "workflow_dispatch" .github/workflows/*.yml
python3 -c "import yaml; yaml.safe_load(open('.github/workflows/daily-collect.yml'))"
```

### 4.4 Git 操作在沙箱环境中

**问题：** 沙箱 Linux 环境中 `git` 命令经常遇到：
- `Committer identity unknown` → 每次 commit 需要显式指定 `-c "user.email=..." -c "user.name=..."`
- `detached HEAD` → `git checkout main` 后手动 cherry-pick
- 网络驱动器上的 `.git` 目录损坏 → 用 `/tmp` 目录做中转，clone 后修改再 push

**推荐工作流：**
```bash
# 在 /tmp 下操作 Git，避免网络驱动器问题
git clone "https://token@github.com/user/repo.git" /tmp/repo
cd /tmp/repo
# 修改文件...
git add -A
git -c "user.email=actions@github.com" -c "user.name=GitHub Actions" commit -m "message"
git push
```

### 4.5 gitignore 与 CI 的冲突

**问题：** `.gitignore` 中排除 `output/*` 是为了本地开发不提交生成文件，但 CI 中需要将这些文件 push 回仓库。

**解决方案：** CI 中使用 `git add -f output/` 强制添加被 gitignore 排除的文件。

## 五、知识存储与记忆管理

### 5.1 项目 CLAUDE.md 的编写

为每个项目编写 `CLAUDE.md`，作为 AI 助手的"项目说明书"。应包含：

```markdown
# CLAUDE.md

## Project overview（项目一句话描述 + 核心数字）
## Architecture（架构图 + 数据流）
## Key modules（模块表格 + 路径 + 用途）
## Configuration patterns（配置文件说明 + 关键设计理念）
## Important implementation details（已知坑点 + 解决方案）
## Local development（本地运行命令）
## Deployment（部署地址 + 工作流说明）
```

### 5.2 经验文档化

**原则：** 完成每个 Phase 后，如果遇到了需要反复排查才解决的坑，立即记录下来。本案例中的关键经验包括：

- 网络驱动器 null 字节问题
- 相对导入 vs 绝对导入
- GitHub Actions schedule 不可靠
- LLM 欠费导致 400/401 的各种错误信息
- gitignore 与 CI force-add 的冲突
